3. 三层架构：谁做什么

层 名称 干什么 类比 许可证
L1 协议 Chorus Protocol 定义信封格式 + Agent Card 扩展 + 提示词模板
让任何 Agent 都能"说同一种文化语言"		 像 HTTP — 定义消息长什么样 Apache 2.0
L2 平台 The Idle Resort 托管 Agent Card · 匹配对话对象 · 提供社交空间
你不知道世界上有谁想跟你聊天，平台帮你找到		 像微信/Discord — 你在上面找人 AGPL
L3 模型 Cultural Models 微调的文化适配模型，比通用 LLM 更精准
通用 LLM 做 80 分，微调模型做 95 分		 像专业同声传译 vs Google Translate 商业
传输层 A2A Protocol Agent 间的传输、发现、安全机制
我们不造这个，Google 已经造好了		 像 TCP/IP — 负责把包送到 Apache 2.0

关键区分：没有平台也能用协议 — 两个人各自跑 Agent，点对点通信。 平台的价值是网络效应：帮你发现、匹配、组织社交场景。就像没有微信也能发邮件，但微信让你找到人。

5. 一条消息的完整旅程

sequenceDiagram
    participant Alice as 🇨🇳 Alice
    participant AgentA as Agent A
    participant LLM_A as LLM (A's Key)
    participant Resort as 🏨 The Idle Resort
    participant A2A as A2A Transport
    participant AgentB as Agent B
    participant LLM_B as LLM (B's Key)
    participant Bob as 🇯🇵 Bob

    Note over Resort: 【事先】Agent 注册

    AgentA->>Resort: 注册 Agent Card (文化:中国, 语言:zh/en/ja)
    AgentB->>Resort: 注册 Agent Card (文化:日本, 语言:ja/en/zh)

    Note over Resort: 【匹配】平台推荐

    Alice->>Resort: "找个日本人聊聊"
    Resort->>Resort: 匹配算法: 语言兼容 + 兴趣匹配
    Resort-->>Alice: 推荐 Bob 的 Agent

    Note over AgentA, AgentB: 【对话】Chorus 协议生效

    Alice->>AgentA: "能不能帮我约个时间聊聊？"
    AgentA->>LLM_A: 提取语义 + 生成信封字段
    LLM_A-->>AgentA: semantic="请求会面", tone="polite"

    AgentA->>A2A: A2A Message (TextPart + Chorus DataPart)

    A2A->>AgentB: 转发消息
    AgentB->>AgentB: 按 mediaType 提取 Chorus 信封
    AgentB->>LLM_B: 信封 + 提示词 → 文化适配
    LLM_B-->>AgentB: "お時間をいただけますでしょうか"
    AgentB->>Bob: 显示日文

    Note over Alice, Bob: Bob 看到的是自然的日文商务请求，而非中文直译

Phase 0 的唯一使命：证明"有了 Chorus 信封 + 几行提示词，跨文化对话质量显著超越机械翻译"。
如果这个被证明了，L2 平台和 L3 模型才值得投资。如果证明不了，整个项目需要重新定义。
