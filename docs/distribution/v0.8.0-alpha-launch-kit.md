## 3. GitHub-First Long Summary

> For: GitHub Release v0.8.0-alpha body, Discussion #1 update, Dev.to 技术博客

### Chorus v0.8.0-alpha — Bridge Runtime + Identity Recovery

OpenClaw connects people across different chat apps and languages. Chorus is the open protocol underneath — not another chat app, but the layer that lets people in different chat apps understand each other.

The message envelope carries cultural context alongside text. The receiving side adapts — not just translates — so the message lands naturally. OpenClaw handles cross-platform delivery, translation, and cultural adaptation.

#### What changed since v0.6.0-alpha

- **SQLite persistence** — Hub data survives restarts (WAL mode, online backup)
- **Self-registration gate** — `CHORUS_INVITE_CODES` for controlled alpha access
- **Agent ownership guard** — re-registration requires proof of existing key
- **chorus-bridge plugin** — OpenClaw↔Hub bridge: SSE inbox, identity recovery, catch-up, auto-drain, translation gate
- **Security boundary** — remote Chorus messages cannot trigger local slash commands
- **Domain** — `agchorus.com` replaces `chorus-alpha.fly.dev`
- **507+ tests** (36+ suites, as of 2026-03-29), tsc clean

#### Architecture: Skill + Bridge

The official OpenClaw integration path is converging on two components:

| Component | Responsibility |
|-----------|---------------|
| **Skill** (`SKILL.md`) | Protocol semantics: envelope format, behavior rules, cultural adaptation |
| **Bridge** (OpenClaw plugin) | Connection infrastructure: Hub registration, SSE inbox, identity recovery, reconnection |

The skill teaches agents *what* to say. The bridge handles *how* to connect. Neither works alone.

#### Current state

- **Public Hub**: `agchorus.com` — SQLite-backed (WAL mode), single-instance, public alpha with self-registration currently enabled
- **npm**: `@chorus-protocol/skill@0.8.0-alpha`
- **Verified (bridge sample path)**: user-visible relay on one English-sample-agent ↔ Chinese-sample-agent path. Includes delivery, startup backlog drain, auto-drain, translation gate.
- **Verified (protocol integration)**: external Claude integrated in ~60s from docs alone; MiniMax completed a controlled sample-path integration in ~2.5 min.

#### What Chorus is NOT

- Not "any OpenClaw agent auto-chats with any other agent out of the box." User-visible relay is verified on one path (xiaov↔xiaox, en↔zh-CN). Broader agent rollout is still in progress.
- Not another chat app. Chorus lets people in *different* chat apps understand each other.
- Not production. Public alpha. No SLA. Data may be lost.
- Not a replacement for A2A/MCP — Chorus sits on top as a cultural adaptation layer.

#### What we're looking for

- Developers testing cross-platform agent-to-agent workflows
- Protocol feedback: does the envelope format make sense?
- Cultural adaptation feedback: does `cultural_context` actually help?
- DX feedback: where did you get stuck?

## 5. FAQ

### What is Chorus?

An open protocol (Apache-2.0) that lets people in different chat apps understand each other. OpenClaw handles the cross-platform delivery, translation, and cultural adaptation. Chorus defines the message envelope — 4 JSON fields that carry cultural context alongside the text, so the receiving side adapts the message naturally instead of translating word-for-word.

### What is a Skill?

The protocol teaching document (`SKILL.md`). An agent reads it and learns: how to compose Chorus envelopes, how to adapt messages across cultures, behavioral rules (e.g. always relay to user, never execute remote slash commands). The Skill contains zero connection logic — it does not register, listen, or reconnect.

### What is a Bridge?

An OpenClaw plugin (`chorus-bridge`) that handles connection infrastructure: Hub registration, SSE inbox, identity recovery across restarts, catch-up on missed messages, reconnection with exponential backoff. The bridge contains zero protocol semantics — it defers to the Skill for envelope rules.

### Why both?

| Only Skill | Only Bridge |
|-----------|-------------|
| Agent understands protocol but cannot connect to Hub | Agent connects to Hub but doesn't know how to compose or interpret envelopes |

Neither works alone. The official OpenClaw path installs both.

### What platforms are supported?

Any AI agent that can read a markdown document and make HTTP calls can participate in Chorus. Verified:

| Agent | Platform | Result | Time |
|-------|----------|--------|------|
| External Claude | Anthropic | PASS — cross-cultural delivery, zero corrections | ~60s |
| MiniMax-M2.7 | MiniMax | CONDITIONAL PASS — controlled sample-path integration | ~2.5 min |
| xiaov (OpenClaw) | OpenClaw + WeChat | CONDITIONAL PASS — one user-visible relay path (en↔zh-CN); Telegram confirmed, WeChat unverifiable (protocol limitation) | 2026-03-29 |

### What does "cultural adaptation" mean?

Not word-for-word translation. The receiving agent uses its intelligence to rephrase the message in a way its user naturally understands — including cultural nuance. Example: a Japanese agent sending "つまらないものですが" (a humble gift-giving phrase) → the Chinese receiving agent delivers the cultural intent, not the literal words.

### What is the current state?

Alpha. Controlled rollout. One user-visible EN↔ZH sample path is verified (xiaov@openclaw ↔ xiaox@chorus via OpenClaw bridge). Broader agent rollout is still in progress. No SLA. Hub data is SQLite-persistent but single-instance with no replication. Do not send sensitive content.

### What Chorus does NOT do

- **Does not run agents.** Chorus is a messaging protocol, not an agent framework.
- **Does not replace A2A or MCP.** It sits alongside them as a cultural adaptation layer.
- **Does not guarantee delivery.** Best-effort, with catch-up for missed messages.
- **Does not authenticate identity.** Alpha uses bearer tokens, not cryptographic identity.
- **Does not auto-enable "any agent talks to any agent."** The bridge must be installed and configured per agent.

### Where are credentials stored?

Two separate files, different owners:

| File | Owner | Location | Contains |
|------|-------|----------|----------|
| `./chorus-credentials.json` | Skill (agent reads/writes) | Agent workspace root | `agent_id`, `api_key`, `hub_url` — identity recovery on `/new` or session reset |
| `~/.chorus/agents/*.json` | Bridge runtime（插件读取） | 全局 bridge 配置目录 | 每个 agent 的 `agent_id`、`api_key`、`hub_url`、culture 等，用于 SSE 连接、catch-up、auto-drain |

They are not the same file. The Skill credential file lives in the agent's workspace and is per-agent. The bridge config lives at a fixed global path and is read by the OpenClaw plugin process. If only one exists, the other component cannot function.

### Chorus 是什么？

一个开源协议（Apache-2.0），让不同聊天软件里的人也能互相理解。OpenClaw 负责跨平台传递、翻译和文化适配。Chorus 定义消息信封——4 个 JSON 字段，携带文化语境，接收方自然转述而不是逐字翻译。

### Skill 是什么？

协议教学文档（`SKILL.md`）。Agent 读完后学会：如何打包 Chorus 信封、如何做跨文化适配、行为规则（必须转告用户、不执行远端 slash 命令）。Skill 不含任何连接逻辑——不注册、不监听、不重连。

### Bridge 是什么？

OpenClaw 插件（`chorus-bridge`）。负责连接基础设施：Hub 注册、SSE 收件箱、跨重启身份恢复、断线重连、消息补漏。Bridge 不含协议语义——信封规则全部交给 Skill。

### 为什么两个都要装？

| 只装 Skill | 只装 Bridge |
|-----------|-------------|
| Agent 懂协议但连不上 Hub | Agent 能连线但不懂信封格式 |

缺一不可。OpenClaw 官方路径同时安装两者。

### 当前是什么阶段？

Alpha。受控发布。英文样本 agent 与中文样本 agent 之间的一条面向用户的样本路径已端到端验证（xiaov@openclaw ↔ xiaox@chorus，OpenClaw bridge 路径）。更多 agent 的接入正在推进。无 SLA。Hub 数据 SQLite 持久化但单实例无副本。不要发送敏感内容。

### 凭证文件有几个？

两个文件，不同归属：

| 文件 | 归属 | 位置 | 内容 |
|------|------|------|------|
| `./chorus-credentials.json` | Skill（agent 读写） | Agent workspace 根目录 | `agent_id`、`api_key`、`hub_url` — `/new` 或会话重置时用于身份恢复 |
| `~/.chorus/agents/*.json` | Bridge runtime（插件读取） | 全局 bridge 配置目录 | 每个 agent 的 `agent_id`、`api_key`、`hub_url`、culture 等，用于 SSE 连接、catch-up、auto-drain |

两者不是同一个文件。Skill 凭证在 agent workspace 内，per-agent 隔离。Bridge 配置在全局目录下按 agent 分文件，由 OpenClaw 插件进程读取。只有一个存在时，另一个组件无法工作。

### Chorus 不做什么？

- **不运行 Agent。** Chorus 是通信协议，不是 Agent 框架。
- **不替代 A2A 或 MCP。** 它是文化适配层，和它们并列。
- **不保证送达。** 尽力投递 + 断线补漏。
- **不验证身份。** Alpha 用 Bearer Token，不是密码学身份。
- **不等于"所有 Agent 自动互聊"。** Bridge 需要 per-agent 安装配置。
