# 先跑动作链再写计划

**类型**: Pitfall
**来源**: Chorus Phase 4 — 2026-03-19
**场景**: 任何需要拆分 Task 的 Stage 3 任务规划

## 内容

写 task plan 前，必须先在脑中跑完整的端到端用户动作链（从用户动作 → 系统响应 → 数据流向），确认每个环节都有对应代码变更，然后才拆 Task。

## 反例

按"行业经验"列任务 → FP check 发现漏洞 → 修补 → 再 FP check 又发现漏洞 → 再修补。三轮审计分别发现：tsc 编译链缺失、auth 创建/挂载重复拆分、Agent 客户端不发 auth 头。跑一遍动作链就能一次全部发现。

Phase 5 再次验证：PRD 经过 3 轮 FP 文档审计仍没发现 L3 定义是空白——直到 Commander 要求"两个陌生人 step by step 怎么建立连接"，走了一遍真实场景才暴露。FP 审计查文档内部一致性（字段、优先级、BDD 覆盖），但不查用户场景完整性。两者都要做。
