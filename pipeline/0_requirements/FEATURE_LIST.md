## 功能详情

### F1: Chorus 语义信封格式规范（协议核心）

> 定义 Agent 之间交换消息的标准信封格式，作为 A2A Message 的扩展。一份规范文档。

| 编号 | 功能描述（业务语言） | 操作类型 |
|------|---------------------|---------|
| F1 | 定义 Chorus 消息信封规范：必填字段（协议版本、发送方原始语义意图、发送方文化背景）+ 可选扩展字段（意图类型标签、正式度、情感基调、关系亲疏）。信封始终保留原始语义，接收方 Agent 可据此做二次判断。 | 规范定义 |

### F2: Agent Card 文化扩展规范

> 扩展 A2A Agent Card，让 Agent 能声明"我代表什么文化背景的用户、我会什么语言"。对话发起方读取对方 Card 确认双方可通信。

| 编号 | 功能描述（业务语言） | 操作类型 |
|------|---------------------|---------|
| F2 | 定义 A2A Agent Card 的 Chorus 扩展字段：用户文化背景、沟通偏好、支持语言列表。发起对话前 Agent 读取对方 Card 确认语言能力匹配。 | 规范定义 |

### F3: 最小文化适配提示词模板

> 协议自带的几行提示词，提醒 Agent 做文化适配而非机械翻译。

| 编号 | 功能描述（业务语言） | 操作类型 |
|------|---------------------|---------|
| F3 | 提供一套最小提示词模板（协议规范的一部分），Agent 实现者须在系统提示中包含，确保 Agent 传达意图而非逐字翻译、适配对方文化习惯、保留原始情感基调。 | 规范定义 |

### F4: 三组对比验证实验

> 用 200 条预制语料（100 条文化禁忌 + 100 条俚语），跑三组对比，用数据回答两个核心假设。

| 编号 | 功能描述（业务语言） | 操作类型 |
|------|---------------------|---------|
| F4 | 实现参考 Agent + 预制 200 条中日测试语料 + 跑三组对比实验：(A) 机械翻译 baseline (B) 极简信封 + 提示词 (C) 完整信封 + 提示词。每条输出由 LLM-as-judge 按三维度打分（意图保留/文化适当性/自然度，各 1-5 分）。B>>A 证明提示词有效（A-05），C>>B 证明结构化字段有价值（A-08），C≈B 则砍掉可选字段。 | CLI 验证实验 |

## Phase 0 追踪说明 (2026-03-18)

### 实现列注释

所有 F-ID 的"实现"标记为 `✅ spike`，表示通过 `spike/` 目录下的一次性实验脚本完成验证，而非 `src/` 目录下的正式 TDD 代码。Phase 0 的交付物是**实验结论**（`results/summary.md`），不是生产代码。

### QA 列注释

Phase 0 的 spike 代码不进入生产，对其做完整 QA 审查无意义。QA 列统一标记为"Phase 0 豁免"。

### F1 验收注释

F1 标记为 `⚠️ 需修订`，原因：实验发现 `cultural_context` 字段是协议核心价值的主要驱动力，但当前 `chorus-envelope.schema.json` 未包含此字段。Phase 1 必须修订 Schema。详见 `INTERFACE.md` 末尾的"Phase 0 实验修正"章节和 `results/summary.md` 的"规范-实验差异声明"。
