### A-02 Risk 评级质疑

**PM 的假设**: A2A Extension 机制足以承载 Chorus 所需字段，Risk=L。
**我的质疑**: PROTOCOL_LANDSCAPE 列出了 A2A 的 6 项"致命缺陷"，意味着 Chorus 要填补的信息量远超一个简单的 KV Extension。如果 Extension 有嵌套深度或大小限制，Chorus 的文化本体可能塞不进去。
**破坏性场景**: Stage 1 Lead 发现 A2A Extension 是 flat KV，无法承载嵌套的 Cultural Context Ontology，被迫 fork A2A Message 格式。Phase 0 的"不从零造"定位崩塌，工作量翻倍。
**建议**: Risk 从 L 提升至 M，并要求在 Gate 0 前完成 A2A Extension 字段的实际结构验证（写一个 5 行代码 spike）。

### A-05 验证方案质疑

**PM 的假设**: "预制测试语料库 200 条，Chorus vs 机械翻译自动对比"能验证假设。
**我的质疑**: "自动对比"需要自动评判标准。文化适配质量不像代码测试有 pass/fail，它本质上是主观的。没有定义评判方法的"自动对比"是空话。
**破坏性场景**: Dev 实现了 200 条对比，输出两列文本，Commander 逐条人工阅读发现某些 Chorus 适配反而不如机械翻译——但没有量化指标，无法说"60% 更好，40% 更差"，假设验证结论模糊不清。
**建议**: 定义最小评判标准——哪怕是"Commander + 1 名日语母语者人工打分 1-5 分"也行。关键是要有预定义的评判流程。

### FP 整改建议（Atomic Plan）

1. **[解决 C1]** 删除 BDD F4.2 Error Case 中的消息队列和断线恢复承诺。替换为 Phase 0 合理的降级行为："Agent A 在等待超时后通知用户'消息发送失败，请重试'"。不保留队列，不承诺恢复。
2. **[解决 C2]** 在 FEATURE_LIST F2.2 描述中明确：Phase 0 的语言能力匹配是**静态读取对方 Agent Card 字段**，不涉及运行时动态协商。将操作类型从"接口调用"改为"Agent Card 字段读取"。
3. **[解决 C3]** 在 PRD A-05 验证方案中定义评判方法：(a) 每条输出由 Commander + 1 名目标语言母语者按 3 个维度打分（意图保留 1-5、文化适当性 1-5、自然度 1-5）；(b) Chorus 平均分 > 机械翻译平均分 = 假设成立。将此评判标准同步到 FEATURE_LIST F4.4 和 BDD F4.4。
4. **[解决 H1]** 将 A-02 Risk 从 L 提升至 M。
5. **[解决 H5]** 在 PRD 非功能性需求表中增加"协议版本标识：信封须包含 version 字段"。
6. **[解决 H7]** 为 F3.1 增加一个 BDD Error Case：文化适配产生不当/刻板印象表达时的降级行为。
