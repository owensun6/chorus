# S-03-12 Burst-200

## Goal

Run a 200-message burst and verify that Bridge v2 pruning matches design:

- prune only oldest prunable records
- keep active records
- do not delete continuity

## Baseline

After `S-03-11`, live `xiaox` state was still below both caps:

- `inbound_facts = 189`
- `relay_evidence = 137`

Because the implementation caps are `500 / 500`, a plain 200-message burst would not reach pruning.

To force pruning without touching repo code, I added controlled old ballast directly into live `xiaox` durable state:

- `+120` finalized inbound facts with very old `observed_at`
- `+1` retryable inbound sentinel: `s0312-keep-retryable`
- `+170` confirmed relay records with very old `submitted_at`

That moved the pre-burst state to:

- `inbound_facts = 310`
- `relay_evidence = 307`
- `relay_unconfirmed = 62`

This setup makes the 200-message burst cross the inbound cap, while keeping one non-prunable inbound record and a stable set of live unconfirmed relays.

## Burst Input

One sequential 200-message burst was sent from `xiaov@openclaw` to `xiaox@chorus`.

- conversation: `s0312-burst-200-20260326`
- marker prefix: `S0312_BURST200_2026-03-26T11:20+08`
- turn numbers: `2001` through `2200`

Raw send record:

- [/tmp/s0312-burst-200.json](/tmp/s0312-burst-200.json)

Observed send result:

- `200 / 200` requests returned HTTP `200`
- all `200 / 200` Hub responses carried `delivery = delivered_sse`
- first trace: `368d11b2-c512-4870-81d7-32af7851cadc`
- last trace: `083b12be-e058-498d-86c5-614168496657`

As in earlier smokes, direct SSE handling on `xiaox` remained contract-broken (`timestamp` missing), so the state proof again depended on restart/catch-up rather than direct SSE application.

## Inbound Prune Proof

After restart/catch-up finished, `xiaox` durable state was:

- `inbound_facts = 500`
- `burst_fact_count = 200`
- `missing_burst_fact_count = 0`
- cursor:

```json
{
  "last_completed_trace_id": "083b12be-e058-498d-86c5-614168496657",
  "last_completed_timestamp": "2026-03-25T16:32:41.093Z"
}
```

Key invariants:

- all 200 burst traces exist in [`xiaox@chorus.json`](/Users/test2/.chorus/state/xiaox/xiaox@chorus.json)
- continuity keys were preserved:
  - `xiaox@chorus:xiaox@chorus`
  - `xiaox@chorus:xiaov@openclaw`
  - `xiaox@chorus:gate3probe-2026-03-25T11-14-03-076Z@chorus`
- retryable sentinel `s0312-keep-retryable` still exists
- `xiaov` state contains `0` burst facts

Oldest finalized ballast was pruned first:

- `s0312-seed-finalized-000` through `s0312-seed-finalized-009` are gone
- `s0312-seed-finalized-010` and `s0312-seed-finalized-011` still exist

This matches the implementation rule in [`state.ts`](/Volumes/XDISK/chorus/src/bridge/state.ts): oldest prunable inbound facts are evicted, but retryable facts are preserved.

## Relay Prune Proof

The 200-message burst itself produced only `62` additional relay rows, so live relay state stopped at:

- `relay_evidence = 369`
- `relay_confirmed = 307`
- `relay_unconfirmed = 62`

That was still below the relay cap, so relay pruning had not yet fired.

To force relay pruning on the live state, I added `+140` more old confirmed relay ballast rows:

- `s0312-seed2-confirmed-000` through `s0312-seed2-confirmed-139`

This lifted relay state to:

```json
{
  "relay_total": 509,
  "confirmed": 447,
  "unconfirmed": 62
}
```

A plain restart did not reduce that number, because the pending unconfirmed retries did not produce any new state write. So I invoked the real pruning implementation directly against the live file:

```bash
npx tsx -e "import { DurableStateManager } from './src/bridge/state.ts'; async function main(){ const m=new DurableStateManager('/Users/test2/.chorus/state/xiaox','xiaox@chorus'); await m.mutate((s)=>s); } main();"
```

Observed before/after from that live call:

```json
{"before":{"relay_total":509,"confirmed":447,"unconfirmed":62}}
{"after":{"relay_total":500,"confirmed":438,"unconfirmed":62,"seed0":false,"seed6":false,"seed7":false,"seed2_000":true}}
```

Key invariants:

- relay count dropped from `509` to `500`
- unconfirmed count stayed `62`
- sample unconfirmed relays still exist and remain `confirmed = false`:
  - `178c79f8-3a2e-4cbf-bbda-dd29199e4b16`
  - `86e3ddc1-1e44-443f-9107-685fe44127db`
  - `00b5cd7e-10f2-4dfe-b702-5619e5e2e800`
- oldest confirmed ballast was evicted first:
  - `s0312-seed-confirmed-000` through `s0312-seed-confirmed-008` are gone
  - `s0312-seed-confirmed-009` and later remain
  - newer ballast such as `s0312-seed2-confirmed-000` and `s0312-seed2-confirmed-139` remain

This matches the relay pruning rule in [`state.ts`](/Volumes/XDISK/chorus/src/bridge/state.ts): only confirmed relays are prunable; unconfirmed relays survive.

## Verdict

`S-03-12 = PASS`

Why:

- a real 200-message burst was accepted by Hub and fully persisted into `xiaox` inbound state
- inbound pruning fired at the `500` cap and removed only the oldest finalized ballast
- the active retryable inbound sentinel survived pruning
- relay pruning was proven on the same live state via the real `DurableStateManager` implementation, dropping `509 -> 500` while preserving all sampled unconfirmed relays
- continuity was not pruned and `xiaov` did not ingest the burst
